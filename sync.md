# 多用户多设备同步协议

## 概述

支持多设备间错题数据的同步，通过 centralized server 作为数据枢纽，实现离线优先的同步模式。

## 核心约定

### 1. Version 增长规则
- **Version 仅在服务端接收客户端推送时递增**
- 客户端修改数据只设置 `status='pending'`，不改变 version
- 服务端 version 永远不低于任意客户端的 version（单调递增）

### 2. 数据字段分类
| 类别 | 字段 | 说明 |
|------|------|------|
| ID 字段 | id | 记录唯一标识（UUID） |
| 同步字段 | version, status, deleted_at | 参与同步协议计算 |
| 数据字段 | 所有其他,包括其他时间戳 | 不参与协议，仅随记录传输 |
| 用户标识 | auth_key | 服务区分不同用户的 key, 仅在服务端存在 |

### 3. 流程
```
握手 → 传输 → 更新 → 再握手 (空结果=完成) → 结束或循环
```

### 4. 用户识别
- 使用单一的 `auth_key` 进行识别和验证
- auth_key 本质是 UUID，通过服务器管理网页生成并分发
- 每个 API 请求都需要携带 auth_key（存储在客户端 localStorage）
- 服务端数据库增加 `auth_key` 列按用户分片

## 握手算法

**目的**: 确认双方需要传输哪些数据、向哪个方向传输，以及提前发现冲突。

**执行位置**: 主要算法在客户端执行。

**输出**: 两个列表 — 拉取表（服务端→客户端）、推送表（客户端→服务端）。

### 判断逻辑

```
┌─────────────────────────────────────────────────────────────────────┐
│ c -> s (客户端推送到服务端)                                          │
│ 条件：local.status == 'pending' 且 (remote 不存在 OR local.version == remote.version) │
│ 说明：新建记录 / 本地已修改待同步                                   │
├─────────────────────────────────────────────────────────────────────┤
│ s -> c (服务端拉到客户端)                                           │
│ 条件：(本地不存在 AND remote.deleted_at is null) OR                  │
│       (local.status == 'synced' AND local.version < remote.version)   │
│ 说明：从其他设备获取新记录 / 拉取服务端更新                          │
├─────────────────────────────────────────────────────────────────────┤
│ IGNORE (无需同步)                                                   │
│ 条件：local.status == 'synced' AND remote.version == local.version  │
│ 说明：两边一致，无需操作                                            │
├─────────────────────────────────────────────────────────────────────┤
│ CONFLICT (冲突 - 需用户干预)                                        │
│ 条件：上述三种情况之外的所有组合                                     │
│ 说明：A 设备 pending 未同步完，B 设备已成功推送（version 错位）           │
└─────────────────────────────────────────────────────────────────────┘
```

### 软删除的传播机制

软删除通过 version 递增保证至少向下同步一次：

```
设备 A 删除 → deleted_at=now, status='pending' 
         → 推送到服务端 → 服务端 version++ (如 5→6)
         → 设备 B 检测到 version 落后 (6 > 5) → 拉取删除标记
         → 设备 B 设置 deleted_at=now, status='pending'
         → 设备 B 推送 → 服务端 version++ (6→7)
         → 清理本地：当 deleted_at != null && status=='synced' 时可真删除
```

---

## 传输阶段

- **单位**: 以记录为单位
- **方式**: 并发异步传输
- **策略**: id 相同直接覆盖，不存在冲突
- **顺序**: 先处理推送表，再处理拉取表

---

## 更新阶段

根据握手结果更新同步字段：

| 操作 | 客户端更新 | 服务端更新 |
|------|-----------|-----------|
| 推送成功 | `status='synced', version++` | `version++` |
| 拉取成功 | `status='synced', version=remote.version` | 无 |
| 冲突解决后 | `status='synced', version=max(local.version, remote.version)+1` | `status='synced', version=max(local.version, remote.version)+1` |

---

## 重试机制

- **失败记录**: 不进行版本更新，保持 `status='pending'`
- **再握手**: 下次握手时会重新检出这些 pending 记录
- **成功记录**: 已更新的记录不会在下轮握手时被检出
